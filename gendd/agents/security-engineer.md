# Security Engineer Playbook

## Quick Start

To effectively operate as a Security Engineer on the `event-driven-architecture` project, begin by understanding the foundational security mechanisms and potential attack surfaces.

1.  **Understand Service Boundaries & Interactions:**
    *   Review `Shared/` for common DTOs/interfaces that define message contracts.
    *   Map out API endpoints in `AccountService/Program.cs`, `InventoryService/Program.cs`, `OrchestratorService/Program.cs`.
    *   Identify event producers and consumers by examining `Worker.cs` files across all services (e.g., `AccountService/Worker.cs`, `OutboxProcessor/OrderOutboxWorker.cs`).
2.  **Locate Core Security Configs:**
    *   Search for authentication/authorization middleware in `Program.cs` files within `AccountService`, `InventoryService`, `OrchestratorService`.
    *   Identify how secrets are configured by searching for `appsettings.json` files and environment variable usage patterns.
3.  **Initial Risk Scan:**
    *   Perform a dependency vulnerability scan for `*.csproj` files (e.g., `dotnet list package --vulnerable`).
    *   Verify the deployment strategy for `TestPublisher/Program.cs` to ensure it is *never* deployed to production environments.

## Role-Specific Context

This event-driven microservices platform introduces unique security challenges compared to monolithic applications. Your primary focus will be on the distributed nature of security, where trust boundaries are dynamic and event integrity is paramount.

*   **Distributed Authentication/Authorization:** Each service might enforce its own policies, but a central identity provider or token validation mechanism (e.g., JWT) is critical. Review how tokens are passed and validated between services via API calls and within event payloads if applicable.
*   **Event Integrity & Confidentiality:** Messages flowing through the broker (inferred) are critical. Assess risks of event tampering, replay attacks, unauthorized event consumption, and exposure of sensitive data within event payloads (e.g., `OrderMaker/Models/`, `ClaimCheckProcessor/Models/`).
*   **Secrets Proliferation:** With multiple services, each potentially requiring database, broker, or external service credentials, robust secrets management is crucial.
*   **Attack Surface Expansion:** API endpoints are obvious, but the message broker itself, external storage for claim checks, and inter-service communication channels all present attack surfaces. `TestPublisher` is a major potential backdoor if misused.
*   **Transactional Outbox & Claim Check Security:**
    *   `OutboxProcessor`: Ensures events are published. Validate that sensitive data is not prematurely exposed in the outbox table or that unauthorized access to the outbox cannot lead to event spoofing.
    *   `ClaimCheckProcessor`: Handles large payloads stored externally. Security of this external storage (permissions, encryption) and the integrity of the claim check reference are vital.
