## Architecture Context Document

### Relevance to This Codebase

Architecture is a paramount concern for the `event-driven-architecture` project due to its explicit design as an event-driven microservices system. This style introduces complexity around service boundaries, communication patterns, data consistency, and operational resilience. Defining clear event contracts, understanding data ownership, managing eventual consistency, and ensuring the reliability of message processing are crucial for the system's correctness and scalability. Changes to service responsibilities, event schemas, or fundamental infrastructure choices (like the message broker or database-per-service pattern) have wide-ranging impacts, making a strong architectural understanding essential for safe and effective development.

### Key Patterns and Conventions

| Pattern Name                    | Where Used                                                                          | Notes                                                                                                                                                                                                                                                   |
| :------------------------------ | :---------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Event-Driven Architecture**   | All services under `services/`, `OutboxProcessor/`, `TestPublisher/`                  | Services primarily communicate by publishing and consuming events via a message broker (implied). This promotes loose coupling and asynchronous processing.                                                                                              |
| **Microservices Architecture**  | `services/AccountService/`, `services/InventoryService/`, `services/OrchestratorService/`, etc. | The system is composed of multiple independently deployable services, each with a focused responsibility. APIs handle synchronous requests, while Workers process asynchronous events.                                                                  |
| **Outbox Pattern**              | `OutboxProcessor/OrderOutboxWorker.cs`, implied in services that publish events | Ensures atomic updates to a database and publishing of corresponding events. Events are first written to an outbox table within the same transaction as business data, then asynchronously published by the `OutboxProcessor`. Critical for data consistency. |
| **Worker Service Pattern**      | `services/*/Worker.cs`, `OutboxProcessor/OrderOutboxWorker.cs`, `ClaimCheckProcessor/`, `MessageReplicator/` | Background services implemented using .NET's `IHostedService` pattern (inferred) that run continuously, typically consuming messages from a queue or performing scheduled tasks.                                                                          |
| **Claim Check Pattern**         | `ClaimCheckProcessor/`, `ClaimCheckProcessor/Models/`                              | (Inferred) Likely used to manage large message payloads. The actual payload content is stored externally (e.g., in a blob storage), and only a reference (the "claim check") is sent via the message broker. The consumer then retrieves the full payload using the claim check. |
| **Shared Library for Contracts** | `Shared/`                                                                           | A central library containing common data contracts, event definitions, and utility code used across multiple services. This ensures consistency but creates a strong compile-time dependency.                                                              |

### Risks and Hotspots

| Risk                                   | Location(s)                                                                       | Severity | Mitigation                                                                                                                                                                                                                               |
| :------------------------------------- | :-------------------------------------------------------------------------------- | :------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`Shared/` Library Changes**          | `Shared/Shared.csproj`, `Shared/**/*.cs`                                        | Medium   | Changes to types or contracts in `Shared/` can introduce breaking changes across multiple services. Requires careful impact analysis, versioning strategy for public types, and coordinated deployments.                                    |
| **`OutboxProcessor/` Failure/Bottleneck** | `OutboxProcessor/OrderOutboxWorker.cs`                                          | High     | If `OutboxProcessor` fails or bottlenecks, event publishing will halt, leading to data inconsistencies and system-wide delays. Critical for eventual consistency. Implement robust monitoring, alerting, and high availability for this service. |
| **Lack of CI/CD and IaC**              | Entire system                                                                     | High     | Manual deployments increase the risk of human error, inconsistent environments, and slow recovery from failures. Invest in automated pipelines for building, testing, and deploying services and infrastructure.                               |
| **Inter-service Communication Contract Changes** | `Shared/` for event definitions, service APIs (e.g., `services/AccountService/Controllers/`) | Medium   | Breaking changes to event schemas or API endpoints can silently disrupt downstream services. Implement schema evolution strategies (e.g., Avro, Protobuf), versioning (major/minor), and robust consumer-driven contract testing.               |
| **Message Broker Single Point of Failure** | Implied external component                                                        | High     | Reliance on an external message broker makes it a critical dependency. Any downtime or performance degradation of the broker will impact all inter-service communication. Ensure message broker high availability, disaster recovery, and robust connection handling in services. |
| **Eventual Consistency Issues**        | All services, particularly those consuming events                               | Medium   | Complexities in distributed systems can lead to temporary or even permanent data inconsistencies if event processing fails or is out of order. Implement idempotency, retry mechanisms, and robust error handling in event consumers.             |

### AI Agent Guidelines

**DO**
*   **Always analyze the impact of changes to `Shared/`:** Before proposing any modification to files within `Shared/`, always perform a detailed impact analysis on all services that reference `Shared.csproj`. These changes can introduce breaking changes across the entire system.
*   **Verify event contract consistency:** When modifying event-related logic, always cross-reference the event definitions in `Shared/` with their usage in both publishing services (e.g., `AccountService` publishing) and consuming services (e.g., `AccountService.Worker`, `OrchestratorService.Worker`).
*   **Consider eventual consistency:** Any modification involving inter-service communication must account for eventual consistency. Do not assume immediate data propagation or synchronous state updates between services.
